# 개발환경 관련 Readme 파일 (필독)

프로젝트명: linguaproject

기간: 11/10~11/28

아래 내용에 기재된 기본 커밋 메시지 작성 매뉴얼을 읽고 반드시 이를 준수한 형태의 커밋 메시지만 작성하기 바랍니다.

---

## 기초 커밋 메시지 구조

커밋 메시지는 **제목(Subject)**, **본문(Body)**, **꼬리말(Footer)** 세 부분으로 구성되며, 각 부분은 빈 줄로 구분합니다.


<type>(<scope>): <subject>
<body>
<footer>

제목(Subject): 필수 항목입니다. 변경 사항을 요약합니다.

본문(Body): 선택 사항입니다. 변경의 상세한 이유와 내용을 기술합니다.

꼬리말(Footer): 선택 사항입니다. 이슈 트래커 ID, 중요 변경(Breaking Change) 사항 등을 기술합니다.

---

### 제목 (Subject)
제목은 Type(Scope): Subject의 형식을 가집니다.
예시: feat(api): 사용자 로그인 엔드포인트 추가



### 타입 (Type)

변경 내용의 성격을 나타내는 필수 항목입니다. 반드시 소문자로 작성합니다.

feat: 새로운 기능 추가
fix: 버그 수정
docs: 문서 수정 (README.md, API 문서 등)
style: 코드 스타일 수정 (포맷팅, 세미콜론 추가 등. 기능 변경 없음)
refactor: 코드 리팩토링 (기능 변경 없이 내부 구조 개선)
test: 테스트 코드 추가, 수정, 삭제
chore: 기타 잡무 (빌드 설정, 패키지 매니저 설정 변경 등)


### 스코프 (Scope)

변경 사항이 영향을 미치는 범위를 나타냅니다. (예: api, auth, ui, common 등)
스코프는 선택 사항이며, 소괄호(())로 감쌉니다.
팀 내에서 합의된 스코프를 사용하는 것을 권장하는데 귀찮으니까 쓰지 맙시다.


### 제목 (Subject)

변경 사항에 대한 간결한 요약입니다.
명령형으로 작성합니다.

Good: 사용자 인증 기능 추가 (O)
Bad: 사용자 인증 기능 추가함 (X)
Bad: Added user authentication (X - 한글/영문은 프로젝트 규칙에 따름)

- 문장의 첫 글자는 대문자로 작성하는 것을 권장합니다. (영문 기준, 한글은 무관)
- 제목 끝에 마침표(.)를 찍지 않습니다.
- 제목은 50자 이내로 간결하게 작성합니다.


### 본문 (Body)

본문은 선택 사항이지만, 제목만으로 설명이 부족할 때 작성합니다.
제목과 본문 사이에는 반드시 한 줄의 빈 줄이 있어야 합니다.

"무엇을" 변경했는지, 그리고 "왜" 변경했는지를 상세히 설명합니다.
**"어떻게(How)"**보다는 **"무엇(What)"**과 **"왜(Why)"**에 초점을 맞춥니다.

각 줄은 72자 이내로 작성하여 가독성을 높입니다. (줄바꿈 사용)


### 꼬리말 (Footer)

꼬리말은 선택 사항이며, 특정 포맷을 가집니다.
본문과 꼬리말 사이에는 반드시 한 줄의 빈 줄이 있어야 합니다.

이슈 트래커 연동: Closes, Fixes, Resolves 키워드를 사용하여 이슈를 닫을 수 있습니다.

Closes #123
Fixes #456, Resolves #789

중요 변경 사항 (Breaking Changes): API 변경, 하위 호환성 문제 등

반드시 BREAKING CHANGE: (혹은 BREAKING-CHANGE:)로 시작합니다.

BREAKING CHANGE: 사용자 인증 API의 응답 형식이 변경되었습니다.
